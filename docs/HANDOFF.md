# UNIVERSAL PROJECT HANDOFF: NameForge (AutoName MVP)

Этот документ предназначен для передачи контекста новой сессии ИИ или новому разработчику.

## 1. Суть проекта
*   **Что это:** Desktop-приложение на **Streamlit** + локальная **SQLite** + **MoySklad REST API**.
*   **Цель:** Автоматизация генерации канонических наименований и описаний автозапчастей. MoySklad ограничен в поиске и хранении сложной применимости (fitment), поэтому NameForge выступает внешним "мозгом" для формирования `name` и `description`, которые затем синхронизируются обратно в облако.
*   **Успех:** Корректно заполненный каталог (~16 000 SKU) с именами, содержащими ключевую применимость для поиска, и описаниями с полным списком совместимости.

## 2. Текущий статус
*   **Этап:** Phase D завершена (Production Ready).
*   **Стабильно:** Импорт из МС, БД SQLite (схема, триггеры, индексы), генератор имен (шаблоны), расчет `source_hash`, управление статусами, UI Очереди и Карточки, пакетный preview, модуль парсинга `src/naming/`.
*   **В процессе:** Использование модуля `src/naming/` для массового обогащения данных из "грязных" имен поставщиков (высокий потенциал для заполнения полей `make`, `model`, `body`).

## 3. Уже реализовано (фиксированные факты)
*   **Архитектура:** Истина (fitment, шаблоны) — в SQLite; Витрина — в MoySklad.
*   **Модуль `src/naming/`:** Парсер имен в формате `{{Категория}} >Применимость< [Спеки] [[Бренд]]`.
*   **Workflow:** Статусы (`new`, `review`, `approved`, `error`, `locked`). Статус `review` ставится только при изменении данных (`candidate_hash != source_hash`).
*   **Безопасность:** `DRY_RUN=True` по умолчанию; блокировка `name_locked=1` защищает имена от перезаписи.
*   **Интеграция:** Автоматическое сопоставление кастомных атрибутов MoySklad через `config/attr_map.json`.

## 5. Ключевые решения и правила (DO NOT BREAK)
*   **SQLite ONLY:** Никаких ORM (SQLAlchemy и пр.). Только стандартная библиотека `sqlite3` и чистый SQL.
*   **Dry Run:** Все операции записи в API должны проверять флаг `settings.dry_run`.
*   **Hash-driven Sync:** Синхронизация в МС (`PATCH`) только при изменении хэша.
*   **English Code / Russian UI:** Код, комменты, докстринги — English. Интерфейс (Streamlit) — Russian.
*   **No Hardcode UUIDs:** Все ID атрибутов МС берутся из `attr_map.json`.

## 6. Блокеры и риски
*   **[RISK] API Limits:** Лимит 50 запросов в пачке, пауза 0.1с. При превышении МС отдает 429.
*   **[NEEDS VALIDATION] Fill Rate:** Крайне низкая заполненность полей `primary_make`, `primary_model` в текущей БД. Требуется подтверждение оператора на массовое обновление из сырых данных.
*   **[BLOCKER] Custom Entities:** Поле "Бренд" в МС — это справочник. PATCH требует `meta`, а не просто строку (решено через `DirectoryCache`).

## 10. Полный контекст
Проект находится в состоянии "Production Ready" после завершения Phase D. 
- **БД:** `data/autoname.db` содержит актуальный слепок каталога. 
- **Код:** `src/product_workflow.py` — сердце бизнес-логики (preview, sync, hash).
- **Инструменты:** `scripts/import_from_ms.py` для синхронизации с облаком, `scripts/setup_ms_attributes.py` для настройки полей. 
- **UI:** `app.py` запускает интерфейс с 6 страницами (Очередь, Карточка, Шаблоны, Алиасы, Синхронизация, Наименования).
- **Важно:** Всегда сверяться с `PROJECT_CONTEXT.md` и `DEVELOPMENT_PLAN.md` перед внесением изменений в логику статусов или структуру хэша. Любые изменения в атрибутах MoySklad должны отражаться в `config/attr_map.json`.
