# Odoo Master Catalog - Project Context & Knowledge Base

Этот документ содержит полную историю архитектурных решений, промптов и алгоритмов, использованных для преобразования "грязного" каталога автозапчастей в структурированную базу данных для Odoo 19.
Используйте этот файл как **контекст (Context Block)** для инициализации новых чатов с ИИ, а также как методичку для работы с новыми прайсами и каталогами.

## 1. PROJECT CONTEXT BLOCK (Для старта новых чатов)

**Проект:** Парсер и нормализатор каталога автозапчастей (МойСклад → Odoo 19).
**Цель:** Преобразовать 15 800+ "грязных" наименований от поставщиков в идеальный мастер-каталог для PIM-системы Odoo. Разложить данные на атомарные части (Базовый тип, Применяемость, Атрибуты) для автогенерации стандартизированных названий по шаблонам и для точного сопоставления в `AutoPrice_Matcher_v2`.
**Стек:** Python (asyncio, aiohttp, pandas), OpenRouter API (LLM Structured Outputs - Gemini 2.5 Pro), JSONL, Power Query (M).

### Текущий статус
- Успешно отработал `production_parser.py`: все 15 800+ товаров прогнаны через LLM батчами.
- Сгенерирован `odoo_master_catalog.jsonl` со строгой структурой.
- Готовы `jsonl_to_xlsx.py` и `power_query_script.m` для разворачивания JSONL в плоскую таблицу с автоматическим переводом 400+ динамических ключей на русский язык и слиянием дублей.

### Ключевые архитектурные решения
1. **Глубокий парсинг через LLM (OpenRouter):** Отказ от RegEx в пользу AI, так как сырые прайсы неструктурированы. LLM извлекает скрытые смыслы на основе контекста группы товара.
2. **Концепция `golden_type` (Идеальный тип товара):** Тип товара формируется без "зашитых" в него характеристик и применяемости.
3. **Атомизация `applicability` (Применяемость):** Марка, модель, годы, двигатель выносятся в отдельный сложный объект/массив.
4. **Асинхронный батчинг + JSONL + progress.json:** Обработка идет по 25 товаров в 5 потоков, результаты сразу пишутся в JSONL. Защита от потери данных при 502/таймаутах API.
5. **Динамические атрибуты с русским маппингом:** ИИ генерирует любые подходящие атрибуты (`color`, `material`), а скрипт-конвертер переводит их на русский (400+ ключей) и объединяет синонимы.
6. **Шаблон сборки названия для Odoo (4-6 частей):** `[Тип товара] + [Место установки] + [Применяемость] + [1-2 Ключевые характеристики] + [Бренд]`.

## 2. Пайплайн Данных (Как мы получили результат)

1. **`raw_catalog.txt`** -> Исходный сырой файл каталога. Содержал обрывы и неструктурированный текст. Был отброшен в пользу CSV.
2. **`my_catalog.csv`** -> Очищенный выгруженный файл (15 800 строк). Ключевое дополнение: наличие столбца **"Группы"** (например, `Салон/Коврики`). Группа передавалась в ИИ как важнейший контекст.
3. **`odoo_master_catalog.jsonl`** -> Итоговый файл, где каждая строка - это валидный JSON-объект. Сгенерирован скриптом `production_parser.py`.
4. **`odoo_master_catalog.xlsx`** -> Плоская таблица для загрузки в Odoo. Сгенерирована скриптом `jsonl_to_xlsx.py` (на базе логики `power_query_script.m`), который динамически развернул атрибуты и перевел их на русский язык.

## 3. Детали AI-Парсинга

*   **Сервис:** OpenRouter API
*   **Модель:** `google/gemini-2.5-pro` (Выбрана за идеальное соотношение цены/качества при сложном извлечении структурированных данных).
*   **Режим работы:** Батчи по 25 товаров. Асинхронные запросы (5 concurrent потоков). Auto-recovery: при падении батча он дробится пополам.

**Использованный Промпт (PROMPT_V2_ADVANCED):**
```text
Ты архитектор PIM-системы Odoo и эксперт по автозапчастям.
Разбери этот массив автозапчастей по методу "Золотой середины", но с максимальной глубиной детализации характеристик и применяемости.

Правила:
1. `id`: ОБЯЗАТЕЛЬНО верни оригинальный ID товара (Код). Это критично!
2. `golden_type`: Базовый тип товара, описывающий суть и главную функцию (например, "Коврик багажника", "Адаптер крепления", "Радиатор"). Без указания авто, цвета, материала.
3. `brand`: Бренд или производитель (выдели из названия или переданного поля).
4. `applicability`: Объект с глубокой детализацией применяемости к авто:
   - `make` (Марка, например "Lada")
   - `model` (Модель/Семейство, например "Priora (2170)", "Samara (2109, 2114)")
   - `generation` (Поколение/Кузов, если указано)
   - `years` (Годы выпуска, если указано)
   - `engine` (Двигатель, если указано, например "1.6 16V")
   - Если товар универсальный (нет привязки к авто), верни `{"is_universal": true}`.
   - Если несколько авто, верни массив таких объектов.
5. `attributes`: Максимально развернутый объект со всеми найденными характеристиками. Распредели их по логичным ключам-столбцам (например, `color`, `material`, `installation_location` (салон/багажник/кузов/ПТФ), `axis` (передняя/задняя), `side` (левая/правая), `size`, `voltage`, `light_type`, `design_features` (массив особенностей)). НИ ОДНА характеристика или особенность из оригинального названия не должна потеряться!

Верни результат в формате СТРОГОГО JSON массива объектов:
[
  {
    "id": "...",
    "original_name": "...",
    "group": "...",
    "golden_type": "...",
    "brand": "...",
    "applicability": {
      "make": "...",
      "model": "...",
      "generation": "...",
      "years": "..."
    },
    "attributes": {
      "color": "...",
      "material": "..."
    }
  }
]
```

## 4. Смежные проекты и заимствованные идеи

При создании архитектуры использовались решения из прошлых проектов пользователя:

1.  **Из проекта `catalog_preprocessor`:**
    *   Заимствована идея **Батчинга (Batching)**. Избежали ошибки передачи 15к строк за раз.
    *   Переиспользована концепция извлечения `canonical_types` (превратилось в `golden_type`), но с исправлением: в старом проекте тип содержал характеристики (`Лампа H7 светодиодная`), в новом проекте тип очищен (`Автолампа`), а характеристики ушли в `attributes`.
2.  **Из проекта `NameForge` (AutoName MVP):**
    *   **Разделение Universal vs Fitment:** Из NameForge взята логика, что для запчастей нужна жесткая применяемость (Make/Model/Engine/Years), а универсальным товарам (масла, химия, коврики) ставится маркер `{"is_universal": true}`, чтобы менять шаблон имени.
    *   **Словари Алиасов:** Вдохновлено `NameForge/config/attr_map.json`. Для конвертации результата JSONL в XLSX мы создали скрипт, который сводит 400+ сгенерированных ИИ ключей (например, `state` и `condition`) к единым русским столбцам (например, `Состояние`).

## 5. Золотое правило формирования наименований для Odoo (Odoo Naming Convention)

Разложенные данные предназначены для сборки по формуле (от 4 до 6 частей):
`[Тип товара (golden_type)] + [Место установки] + [Применяемость] + [1-2 Ключевые характеристики] + [Бренд]`

**Пример сборки в Power Query / Python:**
*Сырое:* `Радиатор охлаждения 2190, 2190FL 15- 1.6 8 / 16 кл; 2192 15- 1.4 / 1.6; KDAC: 1 вентилятор МКПП Паяный Под кондиционер / Без @ LUZAR`
*Разобранное:* 
- `golden_type`: Радиатор охлаждения двигателя
- `applicability`: Lada Granta / Lada Kalina
- `attributes`: МКПП, С кондиционером и без
- `brand`: LUZAR
*Идеальное Odoo имя:* `Радиатор охлаждения двигателя Lada Granta / Kalina МКПП С кондиционером и без LUZAR` 
*(Остальные технические детали загружаются как Attributes/Variants)*.
